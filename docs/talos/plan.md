# Deep Fork Execution Plan

This checklist tracks execution of `deep-fork-plan.md`. Keep it current:

- When a step is complete, check it off and add the commit hash(es) that implemented it.
- Prefer multiple small commits; reference them here rather than batching progress.
- Product/source de-Kubernetes and naming convergence details are tracked in `docs/talos/chubo-product-source-clean-plan.md`.
- Talos/Talosctl debranding execution details are tracked in `docs/talos/talos-reference-elimination-plan.md`.
- For unattended local QEMU fixture runs, set up NOPASSWD sudoers as described in `docs/dev/chubo-os-qemu-devloop.md` (use `sudo -n` + direct script invocation).

Note on build variants:
- `chubo` is the only build tag for the k8s-less OS variant.
- `chuboos` is deprecated and kept only as a compatibility naming wrapper in scripts/targets (talos: 2151c973c, d2b99c04b).

Compatibility sunset target (explicit):
- Target date to remove Talos/Talosctl compatibility shims from active code paths: Friday, March 6, 2026.
- This date is conditional on all debranding gates being green: Phases 2/3/4 in `docs/talos/talos-reference-elimination-plan.md`, plus green local lanes (`make chubo-guardrails`, core QEMU E2E, helper-bundles E2E, opengyoza-quorum E2E, cluster E2E).

Mixed-version support policy (explicit, decided on February 23, 2026):
- Required for current migration gate: mixed-version core (single-node install -> runtime mTLS -> upgrade -> rollback -> support) and mixed-version cluster control-plane coverage.
- Deferred (non-blocking for this gate): mixed worker-path coverage from old installer `v1.13.0-alpha.1-398-g1070f9e22` (fails decode on `spec.modules.chubo.nomad.networkInterface`) and mixed helper-bundles arm64 coverage (blocked on producing a true old arm64 installer artifact).
- Practical source floor for full multi-node module flows in this gate: images that support current `modules.chubo.nomad.networkInterface` schema (the known failing old source is `v1.13.0-alpha.1-398-g1070f9e22`).

Debranding execution mode (explicit, decided on February 20, 2026):
- **Option 1 (active now, default):** compatibility-first debranding. Make `chubo` names primary while keeping `talos` aliases/wrappers for stability.
- **Option 2 (next):** API/proto namespace migration after Option 1 lands cleanly (major-change slice with dedicated compatibility plan and regen).
- **Option 3 (later only):** hard cutover by removing aliases/wrappers in one sweep, gated by Phase 5 compatibility sunset criteria.
- Unless explicitly overridden by product decision, execution order is **1 -> 2 -> 3**.
